# Training Record Endpoints

> [!IMPORTANT]
> This document was generated by reading the current implementation
> (`TrainingController`, its models, and migrations), not from design docs or
> conversations with the original authors. It describes what the code
> actually does today, including quirks and inconsistencies that may not be
> intentional. Treat surprising behavior as a candidate bug to confirm with
> the team, not necessarily an intended contract.

Developer reference for the two "create" endpoints in `TrainingController`:

- `POST /user/{cid}/training/record` — log a training session
- `POST /user/{cid}/training/otsEval` — submit an OTS (Ops Training Standards)
  evaluation form

Source: `app/Http/Controllers/API/v2/TrainingController.php`
(`postNewRecord`, `postOTSEval`), models `App\TrainingRecord`, `App\OTSEval`,
`App\OTSEvalForm`, `App\OTSEvalIndResult`.

Both endpoints accept `application/x-www-form-urlencoded` or `multipart/form-data`
bodies (standard Laravel request input — JSON bodies also work if
`Content-Type: application/json` is set). `{cid}` in the path is the **student's**
VATSIM CID and must match `[0-9]+`.

## Response envelope

Every response is wrapped by the `response()->api()` macro:

```json
{ "data": { ... }, "testing": false }
```

Errors use `generate_error($msg)`, which nests as `data.status = "error"` /
`data.msg = "<message>"`.

> [!WARNING]
> `response()->forbidden()` **always returns HTTP 403**, even for requests with
> no credentials at all. There is no 401 path for these two endpoints, despite
> what the Swagger annotations in the code claim — don't branch client logic
> on 401 vs 403.

## POST `/user/{cid}/training/record`

Log a new training/session record (classroom, live, or sweatbox), optionally
marking it as an OTS (rating exam) attempt.

### Auth

Route middleware: `semiprivate` (tries session/JWT auth, then falls back to an
API key). Authorization logic lives in `TrainingController::canCreate()`:

- **API key**: a facility API key (`apikey` param) valid for the student's home
  facility, **or** for a facility the student is currently visiting, is
  sufficient — no other checks are applied.
- **Session/JWT**: the caller must be Training Staff (Instructor, Mentor, or
  Senior Staff with the "include TA" flag) at the student's home facility, or
  at a facility the student is visiting. **The caller cannot be the student
  themselves** — self-submitted records are always rejected (403).

If neither condition holds, the endpoint returns `response()->forbidden()`
(**403**).

### Path parameters

| Name  | Type    | Description                          |
|-------|---------|---------------------------------------|
| `cid` | integer | CID of the student the record is for |

### Body parameters

| Field           | Type    | Required | Description |
|-----------------|---------|----------|--------------|
| `instructor_id` | integer | conditionally | Instructor's CID. If omitted (or the caller isn't Senior Staff), it's silently overridden to the **authenticated caller's own CID**. When using an API key with no session, this field is effectively required — there is no fallback and the request fails validation. |
| `session_date`  | string  | yes | `YYYY-MM-DD HH:MM` (24h). Parsed with `Carbon::createFromFormat('Y-m-d H:i', ...)` — extra/missing punctuation fails with 400. |
| `position`      | string  | yes | Position worked, e.g. `SEA_APP`, `ZZZ_CTR`. Automatically uppercased. Must match a specific regex [here](https://github.com/VATUSA/api/blob/65ead285809a08be4a26a2ef636bfc37d5504bd8/app/Http/Controllers/API/v2/TrainingController.php#L608) (GitHub markdown doesn't like it). |
| `duration`      | string  | yes | Session duration, `HH:MM`. Stored as `HH:MM:SS`. |
| `movements`     | integer | no | Number of aircraft/movements handled. Must be numeric if present. |
| `score`         | integer | no | Overall score, `1`–`5`. Must be numeric and in range if present. |
| `notes`         | string  | yes | Free-text session notes. Sanitized server-side with HTML Purifier before storage — arbitrary HTML/script tags will be stripped. |
| `location`      | integer | yes | `0` = Classroom, `1` = Live, `2` = Sweatbox. |
| `ots_status`    | integer | no (default `0`) | `0` = Not OTS, `1` = OTS Pass, `2` = OTS Fail, `3` = OTS Recommended. |
| `is_cbt`        | boolean | no (default `false`) | Marks the record as an automated CBT completion. |
| `solo_granted`  | boolean | no (default `false`) | Whether a solo endorsement was granted in this session. |
| `facility`      | string  | no | Facility ID the record is logged under. Defaults to the authenticated user's home facility, or (for API-key requests) the facility that owns the key. |

### Validation & error responses (400)

Checked in order — first failure wins:

1. Missing any of: `instructor_id` (when unauthenticated), `session_date`, `position`, `notes`, `location` → `"Missing fields; see API documentation."`
2. `movements` present but not numeric → `"Invalid number of movements, must be null or an integer."`
3. `score` present but not numeric or not in `1..5` → `"Invalid score, must be null or an integer and between 1-5."`
4. `instructor_id` does not resolve to an existing user → `"Invalid instructor."`
5. `position` fails the regex above → `"Invalid position."`
6. `location` not one of `0`, `1`, `2` → `"Invalid session location. Must be 0, 1, or 2."`
7. `ots_status == 1` and the student **already has** a passing OTS record whose
   `position` contains the same second segment (e.g. another passing `APP`
   record) → `"The controller has an existing, passing Rating exam record for that position type."`
8. `session_date` fails to parse → `"Invalid date; must be YYYY-mm-dd HH:MM."`
9. `duration` fails to parse → `"Cannot edit record. Invalid duration; must be HH:MM."`

A DB save failure returns **500** with `"Unable to save record."`.

> [!NOTE]
> Error message #9 above ("Cannot edit record...") is a copy-paste leftover
> from the shared validation path used by `editRecord` — it applies here too,
> on record *creation*, not just editing.

### Side effects on success

If `ots_status` is `1` (Pass) or `2` (Fail), the endpoint searches the
student's `evaluations` (`OTSEval` rows keyed on `exam_position`,
`result = ots_status == 1`, `exam_date = session_date`'s date) for a match. If
one exists, the new training record and that OTS evaluation are cross-linked
(`training_record.ots_eval_id` and `ots_eval.training_record_id`) automatically —
you do not need to link them yourself via `otsEval`.

### Success response — `200 OK`

```json
{ "data": { "status": "OK", "id": 1042 }, "testing": false }
```

`id` is the new `training_records.id`. (Note: in the automated test
environment, `isTest()` short-circuits the actual DB write and `id` comes back
`null` — this only affects CI, not production.)

### Example

```
POST /api/user/1234567/training/record
Content-Type: application/x-www-form-urlencoded
Cookie: <session>  (or apikey=<key> as a form field / query param)

instructor_id=1000001&session_date=2026-07-01 18:30&position=SEA_APP
&duration=01:15&movements=22&score=4&location=1&ots_status=0
&notes=Solid+scan,+needs+work+on+sequencing.
```

## POST `/user/{cid}/training/otsEval`

Submit a completed OTS evaluation form for a student (used when signing off a
rating exam / promotion evaluation).

### Auth

Route middleware: `private` (session/JWT only — API keys are **not** accepted
on this endpoint, unlike `training/record`). CORS-restricted.

Authorization (in `postOTSEval`, checked directly — there's no shared helper):

- Caller must be authenticated, **and**
- either an **Instructor** at the student's facility, **or**
- a **Mentor** at the student's facility who also holds rating `>= 4` (I3+)
  and is submitting against `form.rating_id == 2` (a mentor can only submit
  the specific S2 promotion form).

Anything else → `response()->forbidden()` (**403**).

### Path parameters

| Name  | Type    | Description             |
|-------|---------|--------------------------|
| `cid` | integer | CID of the student being evaluated |

### Body parameters

| Field        | Type    | Required | Description |
|--------------|---------|----------|--------------|
| `form`       | integer | yes | ID of an `OTSEvalForm` (`ots_evals_forms.id`) — determines which rating this eval is for and which indicators are expected. |
| `position`   | string  | yes | Exam position, e.g. `SEA_APP`. Uppercased on save. Must match `^([A-Z0-9]{2,3})(_([A-Z]{1,3}))?_(GND|TWR|APP|CTR)$` — note this is a **narrower** set than `training/record` (no `DEL`, `DEP`, `FSS`, and no digits in the middle segment). |
| `date`       | string  | yes | Exam date, `YYYY-MM-DD`. |
| `result`     | boolean/int | yes | `1` = Pass, `0` = Fail. Stored as a boolean column; any falsy value other than an explicit `0`/`1` may behave unexpectedly. Cannot be omitted/null. |
| `notes`      | string  | no | Free text. Passed through `nl2br()` before validation/storage. |
| `signature`  | string  | yes | Must be exactly two comma-separated values (e.g. an X/Y signature-pad coordinate pair or similar client-generated token) — `count(explode(',', $signature)) == 2`. |
| `indicators` | array   | yes | Array of result objects, one per **non-header** indicator on the form (see below). |

`indicators[]` item shape:

```json
{ "id": 15, "value": 2, "comment": "" }
```

| Key       | Description |
|-----------|--------------|
| `id`      | `ots_evals_perf_indicators.id` for this indicator. |
| `value`   | `0` = Not Observed, `1` = Commendable, `2` = Satisfactory, `3` = Unsatisfactory. Defaults to `0` if omitted. |
| `comment` | Optional. Empty string is coerced to `null`. |

> [!CAUTION]
> `comment` is read unconditionally in the controller (`$v['comment']`). If
> you omit the key entirely, PHP will emit an undefined-array-key warning on
> the server. Always send at least `"comment": ""`.

The count and identity of indicators must exactly match the form's indicators
where `header_type != 1` (i.e. everything except "Default Header" rows — plain
fields and "Header With Results" rows both need an entry). Fetch the form via
`GET /training/otsEval/{eval}` or the form/indicator endpoints beforehand to
know what's expected — mismatched counts are rejected outright.

### Validation & error responses (400)

Checked in order:

1. `form` doesn't resolve to an `OTSEvalForm` → `"Invalid evaluation form."`
2. Caller not authorized (see above) → **403 Forbidden**, not 400.
3. `form.rating_id` isn't exactly `user.rating + 1`, or `user.promotion_eligible` is falsy → `"The user is ineligible for this evaluation."`
4. `position` fails the regex → `"Invalid position."`
5. `date` fails to parse as `Y-m-d` → `"Invalid date, must be YYYY-mm-dd."`
6. `result` is `null` → `"No result sent."`
7. `signature` missing or not exactly 2 comma-separated parts → `"No signature sent."`
8. `indicators` missing, not an array, or count mismatch vs. the form's indicators → `"Invalid indicator results."`
9. Eval row fails to save → `"Unable to save submission."`
10. Any individual indicator-result row fails to save → the whole `OTSEval` row
    that was just created is **deleted** (rollback-by-hand, not a DB
    transaction) and `"Unable to save submission. <exception message>"` is
    returned. Rows already saved for other indicators in the same request are
    **not** cleaned up — a partial `ots_evals_indicator_results` set can be left
    behind if this happens mid-loop.

### Side effects on success

The endpoint tries to auto-attach the new eval to an existing training record:
it looks for the most recent `TrainingRecord` for this student where
`ots_status` matches the eval result (`1` if passed, else `2`), `position`
contains the eval's second position segment, and `session_date`'s date equals
`date`. If found, that record's `ots_eval_id` is set to the new eval's ID.

### Success response — `200 OK`

```json
{ "data": { "id": 87 }, "testing": false }
```

> [!NOTE]
> The example body above is simplified for readability — this handler calls
> `response()->ok(['id' => ...])`, the same macro `training/record` uses, so
> despite the OpenAPI annotation showing a bare 200 with no schema, the actual
> body shape matches the training-record response: `{status: "OK", id: <int>}`.

### Example

```
POST /api/user/1234567/training/otsEval
Content-Type: application/x-www-form-urlencoded
Cookie: <session>

form=3&position=SEA_APP&date=2026-07-01&result=1
&notes=Cleared+for+S3.&signature=120,340
&indicators[0][id]=15&indicators[0][value]=2&indicators[0][comment]=
&indicators[1][id]=16&indicators[1][value]=1&indicators[1][comment]=Great+scan
```

## Gotchas worth knowing before integrating

> [!WARNING]
> No true 401s. Both endpoints return 403 for missing *and* invalid
> credentials — don't branch client logic on 401 vs 403.

> [!IMPORTANT]
> `training/record` accepts API keys; `training/otsEval` does not.
> Automations that log routine session data can use a facility API key, but
> OTS eval submission always requires a real instructor/mentor session.

> [!NOTE]
> Self-submission is blocked on `training/record` — an instructor cannot log
> their own training session as the student.

> [!NOTE]
> Position regexes differ between the two endpoints. `training/record` allows
> `DEL/GND/TWR/APP/DEP/CTR/FSS` and alphanumeric middle segments;
> `otsEval` allows only `GND/TWR/APP/CTR` and alphabetic middle segments.

> [!WARNING]
> HTML sanitization is inconsistent: `training/record.notes` is run through
> HTML Purifier before storage; `otsEval.notes` is only `nl2br()`'d, not
> purified. Treat `otsEval.notes` as unsanitized user input downstream.

> [!CAUTION]
> Partial-failure cleanup is incomplete on `otsEval` — if an indicator fails
> to save partway through the array, previously-saved indicator rows for that
> submission are left orphaned in the DB even though the parent eval is
> deleted.
